=== Overview

*{app-name}* is a Windows hardening tool written in Rust. It reads a CSV file of rules and applies the corresponding security configurations to a machine, and backs up the existing configuration before changing anything. Administrators use it to bring an air-gapped or off-domain Windows host to a known security state, and to prove that state afterwards.

The project is named after Prometheus, the Titan who brought fire to humanity in Greek mythology. The reference is deliberate: the goal is to give administrators clear control over the security posture of the machines they manage.

The tool runs in three modes. `apply` enforces the rules from a CSV file on the local system. `backup` exports the current configuration described by those same rules into a CSV file, so a previous state can be restored later. `audit` compares the live system against the rules and produces a compliance score, optionally written to a report file.


=== Why it exists

Most existing Windows hardening tooling was built to *assess* compliance, not to *enforce* it. HardeningKitty, for example, is primarily an auditing tool; its apply mode came later and can be fragile because enforcement was never its core concern. {app-name} was built the other way around: applying configurations is the primary job, auditing is a secondary feature that uses the same rule definitions.

The second difference is how it writes policy. Tools of this kind often edit registry keys directly or run raw `secedit` commands, which leaves the changes scattered across files that are hard to review. {app-name} relies on LGPO to edit Local Policies directly. Every change it makes is immediately visible in `gpedit.msc` and can be inspected or reverted by hand, so the hardened state stays reviewable.


=== Usage

{app-name} runs on Windows and must be run from an elevated (administrator) prompt. Build it with `cargo build --release` or use a prebuilt binary, then invoke the desired subcommand. The `apply` mode creates a System Restore Point before changing anything unless you pass `--skip-restore-point`.

----
# Apply the rules in rules.csv to this machine
promethee apply -c rules.csv

# Back up the current configuration described by rules.csv to backup.csv
promethee backup -c rules.csv -o backup.csv

# Audit this machine against rules.csv and write a report
promethee audit -c rules.csv --report --report-file report.csv
----

Increase output detail with repeated `-v` flags, or reduce it with `-q`. File logging is available through `--log` together with `--log-file`.


=== Rules format

Hardening rules are defined in a CSV file. Each row is one rule, described by the following columns:

- *id*: stable identifier for the rule.
- *name*: human-readable name.
- *category*: grouping used for console output.
- *method*: how the rule is applied (see the link:wiki[Wiki] for the full list and per-method semantics).
- *target*: what the method acts on.
- *option1*, *option2*: method-specific parameters.
- *scope*: scope the method operates in.
- *action*: what to do.
- *tags*: comma-separated tags for filtering.

The available methods include local policy via LGPO, `secedit`, advanced audit policy, Windows services, AppX packages, Windows optional features, Windows capabilities, the Windows Firewall, local users and groups, and Software Restriction Policies (SAFER). Refer to the link:wiki[Wiki] for details on each.

=== Status

The tool is usable for its core purpose, with a few items still outstanding:

* [ ] Support the `[Service General Setting]` scope in `secedit`. The current method cannot parse this scope — its fields don't follow the official INI format (no `=` between key and value) — and fails when it is present.
* [ ] Rule filtering by tag.
* [ ] Include raw `registry.pol`, `GptTmpl.inf`, `audit.csv` and `lgpo.txt` files in backups.
